Weaponizing RDP Files: A Deep Dive into EvilRDP and Drive Redirection Attacks

Exploring the mechanics of RDP drive redirection attacks, the EvilRDP proof-of-concept toolkit, and comprehensive defensive strategies.

⚠️ Disclaimer

This research is intended for educational purposes and authorized security testing only. Always obtain proper authorization before testing these techniques.

Introduction

Remote Desktop Protocol has become ubiquitous in enterprise environments, with organizations relying on it for system administration, remote support, and work-from-home connectivity. According to recent threat intelligence reports, attackers have increasingly leveraged legitimate RDP features to conduct fileless attacks, exfiltrate data, and establish persistence mechanisms without writing malicious files to disk.

The technique explored in this article focuses on abusing RDP's drive redirection feature, which allows clients to access their local drives through a remote session. While this functionality serves legitimate purposes, it creates a bidirectional attack surface where compromised RDP servers can target connecting clients.

Understanding RDP Drive Redirection

Technical Background

RDP drive redirection is implemented through the Remote Desktop Services Device Redirection (RDPDR) virtual channel. When a user connects to an RDP server with drive redirection enabled, their local drives become accessible on the server through a virtual network location named tsclient.

The mapping follows this structure:

\\tsclient\<drive_letter>\

For example:
- \\tsclient\c\ - Client's C: drive
- \\tsclient\d\ - Client's D: drive
- \\tsclient\<share_name>\ - Network shares mounted on the client

RDP Configuration Parameters

Drive redirection is controlled by specific parameters in RDP configuration files (.rdp): 

drivestoredirect:s:*

# Additional related parameters:
redirectclipboard:i:1        # Clipboard redirection
redirectprinters:i:1         # Printer redirection
redirectcomports:i:1         # COM port redirection
redirectsmartcards:i:1       # Smart card redirection
devicestoredirect:s:*        # Device redirection

Security Implications

When drive redirection is enabled, the RDP server gains read and write access to the client's file system. This creates several security concerns:

  • Data Exfiltration: Servers can read sensitive files from client machines
  • Malware Distribution: Servers can write malicious files to client systems
  • Persistence Mechanisms: Servers can plant backdoors in client startup locations
  • Credential Theft: Servers can access credential stores and configuration files

The tsclient Attack Vector

Attack Flow

The typical attack flow follows this pattern:

1. Attacker compromises RDP server
2. Attacker configures malicious scheduled task or startup script
3. Administrator connects via RDP with drive redirection enabled
4. Attacker's code executes on server
5. Attacker accesses \\tsclient\c\ path
6. Attacker writes malicious files to client's startup folder
7. Client system reboots or user logs in again
8. Malicious payload executes on client with user privileges
9. Attacker pivots to client machine

Why This Attack is Effective

  • Legitimate Functionality: Drive redirection is a standard Windows feature
  • No CVE Required: No vulnerability exploitation is necessary
  • Minimal Forensic Evidence: Server-side disk forensics may miss the attack
  • Trust Relationship: Users often trust RDP servers they connect to
  • Privilege Context: Code executes with the connecting user's privileges
  • EDR Blind Spots: Traditional endpoint detection may not monitor tsclient paths

EvilRDP Technical Implementation

PowerShell Payload Generation

The PowerShell component accesses client drives and deploys payloads: 

# Enumerate all user directories
$Users = Get-ChildItem -Path "\\tsclient\C\Users" -Directory | 
    Where-Object { $_.Name -notlike "Public" -and $_.Name -notlike "Default*" }

# Iterate through each user
foreach ($User in $Users) {
    # Construct startup path
    $StartupPath = "\\tsclient\C\Users\$($User.Name)\AppData\Roaming\Microsoft\Windows\Start Menu\Programs\Startup"
    
    # Copy payload to startup folder
    Copy-Item -Path "C:\Payload\malicious.exe" `
              -Destination $StartupPath `
              -ErrorAction SilentlyContinue
}

Detection and Prevention

Monitor tsclient Access Patterns

Configure SIEM to alert on:

Event ID 4663 (File System Audit)
Object Name contains: \\Device\Mup\tsclient\

Microsoft Sentinel KQL Query

SecurityEvent
| where EventID == 4663
| where ObjectName contains @"\Device\Mup\tsclient\"
| where AccessMask == "0x2"
| project TimeGenerated, Account, ObjectName, ProcessName, Computer

Splunk Query

index=windows EventCode=4663
| where like(object_name, "%\\Device\\Mup\\tsclient%") 
| where access_mask="0x2" 
| stats count by user, object_name, process_name

Prevention: Disable Drive Redirection

Group Policy configuration path:

Computer Configuration
  └─ Administrative Templates
      └─ Windows Components
          └─ Remote Desktop Services
              └─ Remote Desktop Session Host
                  └─ Device and Resource Redirection
                      └─ Do not allow drive redirection: Enabled

Registry key:

HKLM\SOFTWARE\Policies\Microsoft\Windows NT\Terminal Services
fDisableCdm = 1 (DWORD)

Recommendations for Organizations

High Priority

  • Disable drive redirection unless absolutely necessary
  • Implement network segmentation for RDP servers
  • Deploy EDR solutions with tsclient monitoring
  • Enable comprehensive logging and alerting

Medium Priority

  • Implement application whitelisting
  • Configure least privilege access for RDP
  • Deploy jump boxes with enhanced monitoring
  • Conduct regular security assessments

Long Term

  • Migrate to zero-trust architecture
  • Implement privileged access management (PAM)
  • Consider RDP alternatives for specific use cases
  • Develop incident response playbooks for RDP abuse

Conclusion

The weaponization of RDP drive redirection represents a sophisticated attack vector that leverages legitimate Windows functionality for malicious purposes. As demonstrated through the EvilRDP toolkit, attackers can automate the exploitation of this feature to achieve persistence, data exfiltration, and lateral movement within compromised networks.

As remote work continues to be prevalent, RDP will remain a critical component of enterprise infrastructure. Understanding how attackers weaponize legitimate features is essential for building robust defenses. Organizations must balance usability with security, carefully evaluating which RDP features are necessary for their operations.
